# ADR-003: PhenoMCP Client SDK Architecture

## Status

**Accepted**

## Context

Client SDKs enable applications to connect to MCP servers and consume their capabilities. The client architecture must support multiple programming languages while maintaining consistent behavior and developer experience across platforms.

Key challenges:

1. **Language Diversity**: SDKs needed for TypeScript, Python, Rust, Go, and more
2. **API Consistency**: Similar patterns across language implementations
3. **Type Safety**: Strong typing for all capabilities
4. **Async Patterns**: Different async models per language
5. **Error Handling**: Consistent error representation across languages
6. **Testing**: Easy testing of MCP client code

## Decision

We will implement language-specific SDKs with shared design principles:

### 1. Core Design Principles

**Rationale**: Shared principles enable cross-language consistency.

**Decision Details**:
- Fluent API design across all languages
- Builder pattern for configuration
- Resource disposal/cleanup patterns
- Promise/Future-based async operations
- Structured error types with context
- Request cancellation support

### 2. Transport Abstraction

**Rationale**: Clients work with any supported transport.

**Decision Details**:
- Transport-agnostic client interface
- Transport selection at initialization
- Transport-specific options per implementation
- Automatic transport detection where applicable
- Connection state management
- Reconnection with exponential backoff

### 3. Capability Discovery

**Rationale**: Clients discover server capabilities dynamically.

**Decision Details**:
- Server capability introspection
- Dynamic type generation from schemas
- Cached capability metadata
- Capability version checking
- Graceful degradation for unsupported features
- Type-safe wrappers for discovered capabilities

### 4. Code Generation

**Rationale**: Generated code ensures type safety and consistency.

**Decision Details**:
- OpenAPI/MCP schema as source of truth
- Generated types for requests and responses
- Generated client methods for tools and resources
- Custom decorator/attribute support
- Documentation generation from schemas
- Version-tied generation (schema version = client version)

### 5. Testing Utilities

**Rationale**: Testing MCP clients requires specialized tools.

**Decision Details**:
- Mock server implementation for testing
- Request/response recording and replay
- Stub generation from schemas
- Test fixtures for common scenarios
- Assert helpers for MCP-specific validations
- Integration test harness

## Consequences

### Positive

1. **Developer Experience**: Familiar patterns in each language
2. **Type Safety**: Generated code prevents runtime errors
3. **Productivity**: Fast setup with SDKs
4. **Testing**: Good testability with provided utilities
5. **Maintainability**: Code generation reduces manual work
6. **Documentation**: Generated docs from schemas

### Negative

1. **Complexity**: Maintaining multiple SDKs requires resources
2. **Lag**: New features need implementation in each SDK
3. **Inconsistency**: Subtle differences between language implementations
4. **Dependencies**: SDKs bring dependency trees
5. **Bundle Size**: Generated code may be large

### Neutral

1. **Versioning**: SDK versions tied to MCP protocol versions
2. **Distribution**: Multiple package managers to maintain
3. **Community**: Community SDKs may diverge from official

## Alternatives

### Alternative 1: Single Reference Implementation

**Considered**: Only TypeScript implementation, others use FFI.

**Rejected Because**:
- FFI adds complexity for users
- Performance overhead
- Not idiomatic for target languages
- Debugging difficulties

### Alternative 2: OpenAPI Only

**Considered**: Generate clients from OpenAPI, no custom SDK.

**Rejected Because**:
- Generic clients lack MCP-specific features
- No built-in capability discovery
- Less developer-friendly
- No testing utilities

### Alternative 3: GraphQL Client

**Considered**: Use GraphQL for all client-server communication.

**Rejected Because**:
- Not part of MCP specification
- Adds unnecessary complexity
- Overkill for MCP use case

### Alternative 4: Protobuf/Gateway

**Considered**: gRPC gateway with protobuf definitions.

**Rejected Because**:
- Requires protobuf toolchain
- Less browser-friendly
- Adds dependency complexity
