# ADR-001: PhenoMCP Transport Architecture - Multi-Protocol Communication Layer

## Status

**Accepted**

## Context

The Model Context Protocol (MCP) enables standardized communication between AI models and context providers. PhenoMCP implements this protocol with support for multiple transport mechanisms to accommodate diverse deployment scenarios from local development to distributed cloud architectures.

Key challenges:

1. **Transport Diversity**: Different environments require different transport (stdio, HTTP, WebSocket, gRPC)
2. **Performance**: Latency-sensitive operations need efficient transport
3. **Security**: Authentication and encryption requirements vary by transport
4. **Scalability**: Connection management at scale
5. **Interoperability**: Must work with existing MCP clients and servers
6. **Fallback**: Graceful degradation when preferred transport unavailable

## Decision

We will implement a multi-transport architecture with the following design:

### 1. Transport Abstraction Layer

**Rationale**: Unified interface enables transport flexibility.

**Decision Details**:
- `Transport` trait/protocol defining common interface
- Pluggable transport implementations
- Transport selection at connection time
- Transport negotiation between client and server
- Transport-specific optimizations where applicable

### 2. Supported Transports

**Rationale**: Different use cases require different transports.

**Decision Details**:
- **stdio**: Local process communication, simplest for CLI tools
- **HTTP/1.1**: Widely compatible, good for simple request-response
- **HTTP/2**: Multiplexing for concurrent streams
- **WebSocket**: Full-duplex for streaming and real-time updates
- **gRPC**: High-performance, type-safe, for internal services
- **Unix Domain Sockets**: Low-latency local communication
- **TCP**: Direct socket connection for custom scenarios

### 3. Connection Management

**Rationale**: Reliable connection handling is critical for production.

**Decision Details**:
- Connection pooling for stateful transports
- Automatic reconnection with exponential backoff
- Health checks and connection validation
- Graceful shutdown with in-flight request completion
- Connection multiplexing where supported
- Load balancing for server deployments

### 4. Security Layer

**Rationale**: Security cannot be transport-dependent.

**Decision Details**:
- TLS 1.3 for all network transports
- mTLS (mutual TLS) for service-to-service authentication
- JWT-based authentication for client connections
- Token refresh without connection interruption
- Per-message encryption option for sensitive transports
- Certificate pinning support

### 5. Message Protocol

**Rationale**: Consistent message format across transports.

**Decision Details**:
- JSON-RPC 2.0 as base message format
- Message batching for efficiency
- Streaming support for large responses
- Compression (gzip, zstd) for large payloads
- Request/response correlation via message IDs
- Error standardization across transports

## Consequences

### Positive

1. **Flexibility**: Same MCP implementation works across environments
2. **Performance**: Can choose optimal transport for each scenario
3. **Security**: Consistent security across all transports
4. **Reliability**: Automatic recovery from connection issues
5. **Interoperability**: Standard protocols enable third-party integration
6. **Scalability**: Connection pooling enables high throughput

### Negative

1. **Complexity**: Multi-transport support adds implementation complexity
2. **Testing**: Must test all transport combinations
3. **Configuration**: Transport selection requires understanding tradeoffs
4. **Overhead**: Abstraction layer adds minimal overhead
5. **Debugging**: Cross-transport issues harder to diagnose

### Neutral

1. **Resource Usage**: Connection pooling consumes memory
2. **Monitoring**: Transport-specific metrics needed
3. **Documentation**: Must document transport selection guidance

## Alternatives

### Alternative 1: Single Transport

**Considered**: Support only HTTP/2 as universal transport.

**Rejected Because**:
- CLI tools need stdio for subprocess communication
- Some environments block HTTP
- Cannot leverage transport-specific optimizations

### Alternative 2: gRPC Only

**Considered**: Use gRPC exclusively for all communication.

**Rejected Because**:
- Requires protobuf definitions
- Harder to debug than HTTP
- Proxy configuration complexity
- Not suitable for browser clients

### Alternative 3: WebSocket Only

**Considered**: Use WebSocket for all communication.

**Rejected Because**:
- Overkill for simple request-response
- Proxy/firewall compatibility issues
- Higher resource usage than HTTP

### Alternative 4: Custom Protocol

**Considered**: Design custom binary protocol.

**Rejected Because**:
- No ecosystem tooling support
- Debugging difficulty
- Reinventing solved problems
